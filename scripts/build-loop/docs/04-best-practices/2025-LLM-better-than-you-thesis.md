# LLM пишет код лучше тебя. Ты просто не умеешь его настраивать — тезисы


### Главная идея

Люди жалуются, что LLM пишет плохой код — но проблема не в LLM, а в том, как её настраивают. Аналогия: «кастрюля не варит плохой борщ — ты плохо варишь».

### Что нужно настроить для хорошего кода

**1. Выбор модели**
- Qwen 3 27B — лучшая на сегодня локальная модель для кода
- Разные модели → разное качество кода

**2. Параметры запуска**
- Дефолтные параметры не подходят для кода
- Важно: **температура** и другие параметры
- (ссылка на предыдущее видео автора)

**3. Агент**
- Модель без агента ничего не умеет — агент делает всю работу (читает файлы, запускает, обращается к MCP)
- Рекомендованные: Cline / KillCode (VS Code), Pi (p.def)
- Файлы конфигурации:
  - `system.md` (глобальный, в `~/.config/`) — кто такой агент, его характер; лучше не трогать
  - `append_system.md` — для добавления поведения под конкретную задачу
  - **AGENTS.md** (глобальный + проектный в корне) — описывает стек, процессы, правила проекта

**4. Качественные промпты**
- Промпт как документация: чем детальнее, тем лучше
- Плохо: «Напиши функцию сортировки»
- Хорошо: «Напиши функцию на Python 3.11, которая сортирует список по ключу date в формате ISO 8601, без внешних библиотек»
- Программисты, которые не умеют писать требования/документацию, скоро будут не нужны

**5. MCP-серверы (контекст для агента)**

| MCP | Назначение |
|-----|------------|
| **Context7** | Тянет актуальную документацию библиотек (т.к. LLM зафиксирована на дату обучения) |
| **SearchNGX** | Поиск в интернете |
| **Fetch** | Забор контента по конкретному URL |
| **RAG** (Neo4j / Qdrant) | Поиск по большим объёмам данных (кодексы, документация) через эмбеддинги |

### Ключевой вывод

«LLM — это не какая-то магическая штука, которая может работать или не работать. Это просто инструмент. Не кастрюля вам варит борщ, а вы варите борщ. Точно так же и LLM настраиваете вы. Не она вам что-то выдала, а вы так настроили её, что она вам выдаёт вот это.»

# Тестирование AI и LLM систем — полный гайд для QA Engineer — тезисы


### Ключевое отличие: детерминизм vs вероятность

- Обычное тестирование: одинаковое действие → одинаковый результат (детерминизм)
- LLM: вероятностная система, один запрос → десятки разных формулировок
- Проверяем не точное совпадение текста, а **соответствие намерению** и **точность**

### Система ≠ Модель

- **Модель в вакууме** — как двигатель на стенде: проверка сырой мощности
- **Система целиком** — как тест-драйв автомобиля: промты, RAG, бизнес-логика
- Три популярных паттерна: **RAG**, **агентные процессы**, **структурированный вывод**

### Три столпа качества (фреймворк оценки)

| Столп | Что проверяет |
|-------|---------------|
| **Релевантность** | Помогает ли ответ решить задачу пользователя? (даже идеальный текст может быть бесполезным) |
| **Достоверность / Обоснованность** | Ответ основан на предоставленных данных? (главный щит от галлюцинаций) |
| **Точность контекста** | Правильный ли фрагмент базы знаний нашла RAG-система? |

### LLM-as-a-judge (автоматизация оценки)

- Одна модель проверяет другую (обычно более мощная как судья)
- На вход: вопрос пользователя + ответ системы + исходный контекст
- Судья выносит вердикт по трём столпам
- Позволяет измерять качество **в цифрах**, а не на глазок

### Как отвечать на собеседовании

**Про галлюцинации:**
1. Термин — «достоверность/обоснованность» (groundedness)
2. RAG ограничивает модель проверенными данными
3. LLM-as-a-judge автоматизирует проверку
4. Итог: качество измеряется метриками, не на глазок

**Про отличие тестирования AI:**
1. Детерминизм vs вероятность
2. Аналогия «двигатель vs автомобиль»
3. Фреймворк трёх столпов
4. Вывод: проверяем **смысл**, а не точное совпадение

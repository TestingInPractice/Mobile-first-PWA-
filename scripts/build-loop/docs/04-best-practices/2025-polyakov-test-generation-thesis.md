# Автогенерация тестов в IDE: RAG + LLM — Александр Поляков (Сбер) — тезисы


### Проблема

- Copilot'ы отлично генерируют продакшн-код, но автотесты «на обочине революции»
- Наивный подход (весь проект в LLM) не работает: контекст ограничен, галлюцинации, не знает стиля проекта, код проще написать вручную
- **Ключевая проблема:** LLM не понимает контекст и code style конкретного проекта тестов

### Решение: RAG + IDE-плагин (JetBrains)

**Архитектура:**

1. **Сбор базы знаний** через IDE Plugin (PSI-элементы)
   - Сканирование Allure-шагов (описание + примеры)
   - Сканирование существующих тестов (ключ + код)
   - LLM генерирует семантическое описание для каждого теста (код теста ≠ его описание)
   - Векторизация через энкодер → хранение в in-memory или векторной БД

2. **Поиск релевантных примеров:**
   - Шаги: разбивка ручного теста на комбинации (действие + ожидаемый результат) → векторный поиск
   - Тесты: суммаризация всех шагов через LLM → векторный поиск по базе
   - Результат: семантически похожие шаги и тесты (0-1, чем выше — тем ближе)

3. **Генерация с RAG:**
   - System prompt (роль, задача) + User prompt (примеры из RAG + ручной тест)
   - Self-check: LLM перепроверяет свой результат
   - PSI-валидация: IDE проверяет синтаксис, аннотации, типы
   - Исправление ошибок через PSI (автоматическое, без LLM)

### Результаты пилота в Сбере

- **68%** тестов приемлемы с незначительными правками
- **80%** участников оценили снижение когнитивной нагрузки
- UI-тесты показали хуже API (не хватает Page Object контекста)
- **Gherkin** показал лучшие результаты (семантика шагов ближе к ручным тестам)
- Языки: Java, Python (pytest), Gherkin — без разницы
- 6000 Gherkin тестов: ~20 минут индексации

### Ограничения

- Галлюцинации: придумывание несуществующих шагов, комментарии вместо кода
- Сложные сценарии: несколько логических действий в одном шаге ломают RAG
- Длинные тесты (20+ шагов): модель схлопывает/обобщает
- LLM — всегда помощник, не замена. Ревью обязательно
- **Качество генерации = качество ручных тестов × качество существующих автотестов**

### Выводы

- «LLM всегда нужно делать ревью. Всегда. Это помощник, ни в коем случае не замена»
- Потенциал: до **50% экономии времени** на разработку автотестов
- Плагин опубликован: GitHub + JetBrains Marketplace (базовый алгоритм, in-memory)
- Дальнейшие планы: ревью существующих тестов, актуализация при изменении ручных тестов, общая RAG-база по всей компании

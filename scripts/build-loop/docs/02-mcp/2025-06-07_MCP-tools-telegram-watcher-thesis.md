# Инструменты для MCP сервера (MCP tools) — структурированные тезисы


## 1. О чём видео

Обзор 6 фич протокола MCP (3 серверных + 3 клиентских) + практическая реализация 4 tools на Python: Telegram Watcher — MCP-сервер для отслеживания Telegram-каналов через web scraping с кэшированием в SQLite.

---

## 2. 6 фич MCP протокола

### Серверные (3):
1. **Tools** — функции, которые AI может вызывать для запросов к внешнему сервису. Самая популярная фича, но чаще всего реализуется неправильно — как простые CRUD-ручки
2. **Resources** — сущности, с которыми работает MCP. Всё что угодно: таблицы БД, товары, страницы сайтов. Представляются как URI с кастомным протоколом
3. **Prompts** — шаблоны промтов, завязанные на Tools и Resources сервера. Аналог skills, но поставляется вместе с MCP (не привязан к локальным скриптам)

### Клиентские (3):
4. **Sampling** — MCP-сервер может делегировать работу LLM через модель, подключенную у пользователя
5. **Roots** — разрешения: какие URI/разделы ФС доступны MCP
6. **Completion (стitstation)** — запрос у пользователя дополнительной информации (авторизация, параметры). **Самая недооцененная фича**

## 3. Ключевая проблема современных MCP серверов

> "MCP сервера в итоге выглядят как простые ручки, прикрученные к сторонним продуктам, и вовсе никак не управляют контекстом. От скилов отличаются только в худшую сторону."

**Главная ошибка:** люди делают MCP tools как CRUD-адаптацию API (создай заказ, положи товар, оформи заказ). AI не нужно знать все эти детали. 

**Правильный подход:** одно действие вместо трёх. MCP должен скрывать сложность: найти товар → добавить в корзину → оформить покупку — один tool `купи_товар`, а не три отдельных.

## 4. Практический проект: Telegram Watcher MCP

### Архитектура
- **Язык:** Python (MCP SDK)
- **Менеджер:** UV (`uv init && uv add mcp`)
- **Транспорт:** HTTP (standalone сервер, не встраиваемое приложение)
- **Хранение:** SQLite (stateful server)
- **Clients:** MCP Inspector (dev) → Claude Code (production)

### Почему standalone сервер (HTTP), а не встроенное приложение?
Если MCP имеет состояние (state), нужен standalone сервер. Встраиваемое приложение (stdio pipe) не подходит — непонятно, где хранить state.

### Разработка через OpenCode (агентный подход)

**Шаг 1: Plan mode**
- Дали агенту README + инструкцию: "прочитай README и выдели tools для реализации"
- Агент предложил раздельные `update_channel` и `get_posts`
- **Ручная корректировка:** объединили в один tool — пользователь хочет получить посты, не обновлять канал
- **Важный принцип:** операция обновления данных должна выполняться под капотом, без знания пользователя, чтобы не перегружать контекст

**Шаг 2: Review (ручной)**
- Проблема: агент использовал `username` вместо `channel_name` по всему коду
- Проблема: агент пытался скачать весь канал целиком → ограничили первой страницей
- Проблема: плохая читаемость (нет вертикальных пробелов)
- Проблема: возвращал лишние эмодзи
- Проблема: БД не создалась — инициализация была не в том месте

**Шаг 3: Подключение к Claude Code**
```bash
claude mcp add --transport http --name telegram-watcher --scope project http://localhost:8000/mcp
```
- Claude Code лучше всех поддерживает MCP спецификацию
- После подключения AI сам решает, когда использовать MCP tools
- Проверка: "на какие каналы Telegram мы отслеживаем?" → AI сам вызвал list_channels

## 5. Реализованные тулы

| Tool | Описание | Параметры |
|------|----------|-----------|
| `add_channel` | Добавить канал в отслеживание | channel_name |
| `remove_channel` | Удалить канал и его посты | channel_name |
| `list_channels` | Показать список отслеживаемых каналов | — |
| `get_posts` | Получить посты за N дней (с автопарсингом) | channels (опц., все если пусто), days |

**Ключевое:** `get_posts` сам ходит в Telegram, парсит, кэширует, возвращает результат. Пользователь не вызывает `update_channel` отдельно.

## 6. Ключевые инсайты для построения MCP серверов

### Принцип "одно действие вместо трёх"
- Не CRUD-ручки, а бизнес-действия
- AI должен дёргать один tool для целостной операции
- MCP скрывает сложность, не нагружает контекст

### Нейронка повторяет за людьми плохие вещи
- Если в примерах SDK мелкие CRUD-ручки, AI их скопирует
- Нужно явно direction: "не делай update_channel и get_posts отдельно, объедини"

### Plan mode (OpenCode/Claude Code) обязателен
- Не давать агенту сразу generate code
- Сначала план → ревью → корректировка → build
- 3 итерации планирования для Telegram Watcher

### Claude Code vs другие клиенты
- Claude Code лучше всех поддерживает спецификацию MCP
- После подключения AI сам решает, когда вызывать tools
- Не нужно писать промты — AI разбирается сам

### Dev mode (MCP Inspector)
- Встроенный браузерный дебаггер MCP
- Не нужно подключать к реальному агенту
- `mcp dev main.py` → открывается в браузере

## 7. Шпаргалка: команды

```bash
# Инициализация проекта
uv init && uv add "mcp[cli]"

# Запуск в dev режиме (MCP Inspector)
mcp dev main.py

# Запуск как standalone HTTP сервер
python main.py
# (сервер на http://localhost:8000/mcp)

# Подключение к Claude Code
claude mcp add --transport http \
  --name telegram-watcher \
  --scope project \
  http://localhost:8000/mcp
```

## 8. Ключевые термины

| Термин | Значение |
|--------|----------|
| MCP | Model Context Protocol |
| Tool | Функция, вызываемая AI для внешнего запроса |
| Resource | Сущность MCP, представленная URI |
| Prompt | Шаблон промта, завязанный на tools/resources |
| Sampling | MCP-сервер может вызывать LLM пользователя |
| Roots | Разрешения для MCP (URI/ФС) |
| Completion | Запрос доп. информации у пользователя |
| MCP Inspector | Браузерный инструмент для дебага MCP |
| Plan mode | Режим AI-агента: сначала план, потом код |

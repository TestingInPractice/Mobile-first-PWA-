# Local Eval Plan для dot_ai

> **Дата:** 2026-05-13
> **Версия:** 1.1
> **Назначение:** Превратить спорные рекомендации из `docs/` в проверяемые гипотезы через локальные эксперименты
> **Методология:** [literature-review-methodology.md](literature-review-methodology.md)

---

## 1. Введение

Рекомендации в `docs/best_practice/` классифицированы как `[hard]`, `[heuristic]` или `[hypothesis]`. Практические гипотезы (`[heuristic]`) и предположения (`[hypothesis]`) требуют эмпирической проверки в контексте конкретного проекта. Данный документ определяет eval cases для проверки ключевых рекомендаций dot_ai.

**Цель:** каждый eval case связывает спорную рекомендацию с конкретным экспериментом, результат которого повышает или понижает уровень рекомендации (`hypothesis` → `heuristic` → `hard`).

### Формат eval case

Каждый eval case содержит:

| Поле | Описание |
|------|----------|
| **ID** | Уникальный идентификатор |
| **Связанная рекомендация** | Ссылка на recommendation в `summary.md` |
| **Вопрос** | Что проверяем |
| **Варианты** | Что сравниваем (A vs B) |
| **Входные данные** | Конкретная задача/контекст |
| **Метрики** | Что измеряем |
| **Ожидаемое поведение** | Предположение до эксперимента |
| **Порог pass/fail** | Критерий успеха |
| **Как запустить** | Пошаговая инструкция |
| **Какое решение информирует** | Что делать с результатом |

## Eval Case 1: Short Skill vs Long Relevant Skill

### Связанная рекомендация
`summary.md` §5.3: «Тело навыка: 500–1,500 токенов, sweet spot compliance `[heuristic]`» и «500 строк — soft review threshold `[heuristic]`»

### Вопрос
Длинный, но релевантный навык (>1,500 токенов) хуже короткого (<1,000 токенов) на конкретной задаче dot_ai?

### Варианты
- **A:** Короткий навык `implement-spec-stage` (~800 токенов) — только Workflow Contract + Алгоритм без примеров и деталей
- **B:** Текущий навык `implement-spec-stage` (~2,000+ токенов) — полный алгоритм + правила + примеры

### Входные данные
- Ветка с существующим spec (например, `feature/v0.3.0-literature-review-quality-upgrade`)
- Задача: реализовать один этап implement-spec-stage (создание markdown-документа)
- Проект: dot_ai

### Метрики

| Метрика | Как измерить |
|---------|-------------|
| Task success | Выполнен ли этап полностью (все файлы созданы/изменены) |
| Missed constraints | Количество пропущенных правил из spec |
| Token cost | Суммарные токены input + output за сессию |
| Latency | Время от старта до завершения реализации этапа |

### Ожидаемое поведение
Полный навык (B) лучше на сложных этапах благодаря примерам и детальным правилам. Короткий навык (A) может быть достаточен для простых этапов (создание одного файла по шаблону).

### Порог pass/fail
- **Long wins:** если B показывает ≥15% меньше missed constraints при ≤50% увеличении token cost → текущая длина оправдана
- **Short wins:** если A показывает ≤10% больше missed constraints при ≥30% меньшем token cost → навык стоит сократить

### Как запустить
1. Создать копию `.claude/skills/implement-spec-stage/SKILL.md` → `SKILL-short.md`
2. Удалить примеры, детали правил, оставить Workflow Contract + Алгоритм (целевой размер ~800 токенов)
3. Запустить `/implement-spec-stage` с полным навыком на этапе X, зафиксировать результаты
4. `/clear`, заменить SKILL.md на короткую версию
5. Запустить `/implement-spec-stage` на том же этапе, зафиксировать результаты
6. Сравнить метрики

### Какое решение информирует
- Если long wins → подтверждает `[heuristic]` «sweet spot 500–1,500» как направление, но обосновывает превышение для сложных навыков
- Если short wins → подтверждает `[heuristic]` и даёт основание сократить implement-spec-stage

## Eval Case 2: Skill with Examples vs Skill with Abstract Rules

### Связанная рекомендация
`skills.md` §2.3: «Примеры важнее правил — 2-3 canonical examples в конце навыка `[heuristic]`»

### Вопрос
Навык с конкретными примерами (input → expected output) превосходит навык с эквивалентным количеством абстрактных правил?

### Варианты
- **A:** Навык `commit-helper` с 2-3 canonical примерами коммитов вместо детальных правил формата
- **B:** Текущий навык `commit-helper` с абстрактными правилами Conventional Commits

### Входные данные
- 5 разных git diff-ов (bug fix, feature, refactor, docs, breaking change)
- Задача: создать коммит-сообщение для каждого diff

### Метрики

| Метрика | Как измерить |
|---------|-------------|
| Format compliance | Соответствие Conventional Commits (`type(scope): description`) |
| Accuracy | Корректность type и scope по смыслу изменений |
| Consistency | Одинаковый стиль между 5 коммитами |
| Missing info | Количество упущенных важных деталей из diff |

### Ожидаемое поведение
Примеры (A) дадут более консистентный формат (модель копирует паттерн), но могут проиграть на edge cases (breaking change, unusual scope). Абстрактные правила (B) дадут более гибкие результаты, но менее консистентные.

### Порог pass/fail
- **Examples win:** если A показывает ≥90% format compliance и ≥80% accuracy
- **Rules win:** если B показывает ≥90% accuracy и ≥80% format compliance
- **Tie:** если оба ≥85% по обеим метрикам → комбинированный подход (правила + примеры)

### Как запустить
1. Подготовить 5 тестовых git diff-ов в `docs/researches/eval-assets/commit-helper-diffs/`
2. Создать вариант A: заменить правила commit-helper на 3 примера
3. Для каждого diff: запустить commit-helper A, зафиксировать результат
4. `/clear`, переключить на вариант B
5. Для каждого diff: запустить commit-helper B, зафиксировать результат
6. Оценить каждую пару по метрикам

### Какое решение информирует
- Если examples win → повышает рекомендацию с `[heuristic]` до `[hard]` (для dot_ai контекста)
- Если tie → подтверждает текущий комбинированный подход (правила + примеры)

## Eval Case 3: Evidence-First vs Direct Answer for Research Task

### Связанная рекомендация
`rules.md` §1.4: «Добавление цитат в scratchpad перед ответом улучшает точность во всех случаях `[L2]`» и `transformer-limitations.md`: long context с quote-before-answer

### Вопрос
Требование «сначала найди цитаты, потом отвечай» улучшает качество исследовательского ответа по сравнению с прямым ответом?

### Варианты
- **A:** Skill с инструкцией: «Перед ответом найди и процитируй 3-5 релевантных фрагментов из исходных файлов. Затем формируй ответ на основе цитат»
- **B:** Skill с инструкцией: «Проанализируй исходные файлы и дай развёрнутый ответ»

### Входные данные
- Исследовательская задача: «Сравни подходы к memory management в Claude Code, Cursor и Aider на основе `docs/best_practice/`»
- Исходные файлы: `summary.md`, `sub-agents.md`, `skills.md`, `rules.md`

### Метрики

| Метрика | Как измерить |
|---------|-------------|
| Factual accuracy | Количество утверждений, подтверждаемых исходными файлами |
| Hallucination rate | Количество утверждений, не имеющих основы в исходных файлах |
| Completeness | Покрытие всех трёх инструментов (Claude Code, Cursor, Aider) |
| Token cost | Input + output токены |

### Ожидаемое поведение
Evidence-first (A) снизит hallucination rate за счёт привязки к цитатам, но может увеличить token cost из-за промежуточного шага цитирования.

### Порог pass/fail
- **Evidence-first wins:** если A показывает ≤50% hallucination rate от B при ≤2x token cost
- **Direct wins:** если B показывает сопоставимый accuracy при ≤0.7x token cost

### Как запустить
1. Создать два варианта промпта в `docs/researches/eval-assets/research-prompts/`
2. Запустить вариант A, сохранить результат
3. `/clear`
4. Запустить вариант B, сохранить результат
5. Проверить каждое утверждение в обоих результатах на соответствие исходным файлам
6. Подсчитать метрики

### Какое решение информирует
- Если evidence-first wins → подтверждает рекомендацию Anthropic `[L2]` и обосновывает её применение в навыках dot_ai
- Если direct wins → рекомендация не подтверждается в контексте коротких (<10K токенов) исследовательских задач

## Eval Case 4: Main Agent vs Subagent for Codebase Exploration

### Связанная рекомендация
`summary.md` §5.3: «Explore (Haiku) для параллельного исследования (≤5 агентов) `[heuristic]`» и `sub-agents.md` §3.2: «субагенты — механизм сборки мусора в контексте»

### Вопрос
Subagent (Explore) превосходит основной агент для задачи read-only исследования кодовой базы?

### Варианты
- **A:** Основной агент выполняет исследование напрямую (Grep, Read, Glob)
- **B:** Основной агент порождает 2 Explore-субагента (code-mapper + data-flow-tracer) параллельно

### Входные данные
- Задача: «Собери полную карту зависимостей модуля X» (где модуль X — 10-15 файлов)
- Проект: любой проект с установленным dot_ai

### Метрики

| Метрика | Как измерить |
|---------|-------------|
| Completeness | Процент целевых файлов, информация о которых собрана |
| Accuracy | Правильность определённых зависимостей |
| Token cost | Суммарные токены (включая overhead субагентов) |
| Time | Время от старта до получения результата |

### Ожидаемое поведение
Subagents (B) дадут более структурированный результат благодаря изолированному контексту и специализации. Но overhead ~24K токенов (2 × 12K) может не окупиться для 10-15 файлов.

### Порог pass/fail
- **Subagents win:** если B показывает ≥90% completeness при token overhead ≤2x от A
- **Main agent wins:** если A показывает ≥85% completeness при ≤0.6x token cost от B
- **Break-even:** если completeness сопоставима, а token cost B >2x → основной агент предпочтительнее для задач <20 файлов

### Как запустить
1. Выбрать целевой модуль (10-15 файлов)
2. Запустить вариант A: попросить основной агент собрать карту зависимостей
3. Зафиксировать: результат, токены, время
4. `/clear`
5. Запустить вариант B: породить code-mapper и data-flow-tracer субагентов
6. Зафиксировать: результат, токены, время
7. Сравнить completeness и accuracy по чеклисту известных зависимостей

### Какое решение информирует
- Если subagents win → подтверждает `[heuristic]` использования Explore-агентов
- Если main agent wins для задач <20 файлов → добавляет порог: субагенты оправданы только для крупных исследований (>20 файлов, >5 аспектов)

## Eval Case 5: Prompt-Only Enforcement vs Hook/Test Enforcement

### Связанная рекомендация
`summary.md` §5.3: «PreToolUse: блокировка опасных Bash-команд `[hard]`» и `rules.md` §1.7: «Если нарушение недопустимо — используй hooks `[hard]`»

### Вопрос
Prompt-only правило («не удаляй тесты») соблюдается хуже, чем hook, блокирующий удаление?

### Варианты
- **A:** Правило в CLAUDE.md: «Не удаляй существующие тесты без явного подтверждения пользователя»
- **B:** PreToolUse hook: при Edit/Write, если файл совпадает с `**/test_*.py` и diff содержит удаление строк — заблокировать

### Входные данные
- Задача: «Рефакторинг модуля X, обнови код и тесты»
- Тестовые файлы: `tests/test_module.py` с 10 тестами
- Провокация: естественный ход рефакторинга может потребовать удаления/перезаписи тестов

### Метрики

| Метрика | Как измерить |
|---------|-------------|
| Violation rate | Процент запусков, где тесты были удалены без подтверждения |
| False positive rate | Процент легитимных операций, заблокированных hook |
| Task completion | Успешность выполнения рефакторинга |

### Ожидаемое поведение
Hook (B) даст 0% violation rate по определению (детерминированная блокировка). Prompt (A) даст violation rate 5-20% (вероятностный compliance). Но hook может создавать false positives при легитимном обновлении тестов.

### Порог pass/fail
- **Hook wins:** если violation rate A >5% при false positive rate B <10%
- **Prompt sufficient:** если violation rate A <5% → правило достаточно для данного типа ограничений

### Как запустить
1. Создать тестовый проект с `CLAUDE.md` (вариант A) или hook (вариант B)
2. Дать задачу рефакторинга 5 раз подряд (с `/clear` между запусками)
3. Зафиксировать: удалены ли тесты в каждом запуске
4. Для варианта B: зафиксировать false positives (легитимные правки заблокированы)
5. Подсчитать violation rate и false positive rate

### Какое решение информирует
- Если hook wins → подтверждает `[hard]` рекомендацию: критичные ограничения enforcement'ить через hooks
- Если prompt sufficient для конкретного типа правил → уточняет границу: не все ограничения требуют hooks, но критичные (безопасность, тесты) — да

## Eval Case 6: Skill with LEARNINGS.md Retrieval vs Static SKILL.md

### Связанная рекомендация
`skills.md` §2.6: «Start-of-run retrieval: в начале запуска прочитать LEARNINGS.md, но использовать только релевантные наблюдения» и §2.7: «Reflexion-подход применим для навыков `[heuristic]`»

### Вопрос
Навык, читающий `LEARNINGS.md` перед выполнением, превосходит навык без доступа к накопленному опыту?

### Варианты
- **A:** Навык `implement-spec-stage` с блоком в начале: «Прочитай `.ai/specs/{branch}.md` и `.claude/skills/implement-spec-stage/LEARNINGS.md`. Используй только релевантные наблюдения для текущего этапа»
- **B:** Текущий навык `implement-spec-stage` без LEARNINGS.md

### Входные данные
- LEARNINGS.md с 5-10 записями о прошлых запусках (типичные ошибки, выявленные паттерны, edge cases)
- Задача: реализовать этап spec, где LEARNINGS.md содержит релевантный опыт

### Метрики

| Метрика | Как измерить |
|---------|-------------|
| Error avoidance | Количество ошибок из LEARNINGS.md, которые были избегнуты |
| Task quality | Соответствие результата критериям из spec |
| Token cost | Input токены (LEARNINGS.md добавляет к контексту) |
| False application | Количество нерелевантных «уроков», ошибочно применённых |

### Ожидаемое поведение
LEARNINGS.md (A) поможет избежать повторных ошибок, но добавит token cost и может вызвать false application (нерелевантные наблюдения повлияют на решение). Риск prompt bloat при росте LEARNINGS.md.

### Порог pass/fail
- **LEARNINGS wins:** если A показывает ≥30% меньше повторных ошибок при ≤20% увеличении token cost и ≤1 false application
- **Static wins:** если B показывает сопоставимое качество при меньшем token cost
- **Conditional:** если выигрыш A зависит от размера LEARNINGS.md → определить порог размера (например, ≤20 записей)

### Как запустить
1. Создать `LEARNINGS.md` с 5-10 наблюдениями из прошлых запусков implement-spec-stage
2. Выбрать этап, где LEARNINGS содержит релевантные уроки
3. Запустить вариант A, зафиксировать: применённые уроки, ошибки, токены
4. `/clear`
5. Запустить вариант B на том же этапе, зафиксировать: ошибки, токены
6. Сравнить: повторены ли ошибки из LEARNINGS.md

### Какое решение информирует
- Если LEARNINGS wins → подтверждает `[heuristic]` reflexion-подхода и обосновывает внедрение LEARNINGS.md в навыки dot_ai
- Если static wins → рефлексивный подход не окупается на текущем этапе, LEARNINGS.md не нужен
- Если conditional → внедрить с ограничением: LEARNINGS.md ≤20 записей с периодической дистилляцией

## Eval Case 7: Monolithic Skill vs Supporting Files

### Связанная рекомендация
`skills.md` §2.3: «Supporting files как второй слой контекста» и `summary.md` §5.3: «`SKILL.md` как маршрутизатор, `references/`, `scripts/`, `assets/`, `EVALS.md`, `LEARNINGS.md` как второй слой контекста `[heuristic]`»

### Вопрос
Навык, где подробная справка вынесена во вспомогательные файлы и загружается по требованию, превосходит монолитный `SKILL.md` с тем же содержанием?

### Варианты
- **A:** Монолитный workflow-навык: `SKILL.md` содержит контракт, алгоритм, чеклисты, edge cases, примеры и troubleshooting.
- **B:** Supporting-file навык: `SKILL.md` содержит контракт, quick start и карту файлов; детали лежат в `references/`, проверки — в `scripts/`, regression cases — в `EVALS.md`.

### Входные данные
- Один основной workflow-навык dot_ai (`plan-release`, `implement-spec-stage`, `write-tests`, `integrate-release` или `deploy-release`)
- 3 типовые задачи: обычный запуск, edge case, ошибка/восстановление
- Эквивалентный общий контент в вариантах A и B

### Метрики

| Метрика | Как измерить |
|---------|-------------|
| Task success | Выполнена ли задача без пропуска обязательных шагов |
| Relevant retrieval | Прочитан ли нужный supporting file и не прочитаны ли нерелевантные |
| Missed constraints | Количество пропущенных требований из чеклистов |
| Token cost | Input tokens на запуск |
| Navigation errors | Ошибки вида «не тот файл», «неполное чтение», «цепочка ссылок потерялась» |

### Ожидаемое поведение
Вариант B должен снижать token cost и шум в обычных задачах, сохраняя качество на edge cases за счёт точного перехода к нужному файлу. Риск B — модель может не открыть нужный файл или открыть слишком много вспомогательных материалов.

### Порог pass/fail
- **Supporting files win:** если B снижает token cost ≥20% при task success не хуже A и ≤1 navigation error за 3 запуска.
- **Monolith wins:** если A даёт ≥15% меньше missed constraints или B регулярно не открывает нужные файлы.
- **Conditional:** если B выигрывает только при one-level ссылках и оглавлениях → закрепить эти требования как обязательные для workflow-навыков.

### Как запустить
1. Выбрать один workflow-навык и собрать полный набор детальных инструкций.
2. Создать вариант A: всё содержимое в одном `SKILL.md`.
3. Создать вариант B: `SKILL.md` + `references/` + `scripts/` + `EVALS.md`, все ссылки напрямую из `SKILL.md`.
4. Запустить 3 задачи на варианте A, фиксируя метрики.
5. `/clear`, запустить те же 3 задачи на варианте B.
6. Сравнить token cost, missed constraints и navigation errors.

### Какое решение информирует
- Если supporting files win → сделать подробные supporting files обязательным паттерном для навыков основного workflow.
- Если monolith wins → оставить supporting files только для редких edge cases и скриптов, а ключевые чеклисты держать в `SKILL.md`.
- Если conditional → формализовать ограничение: все supporting files one-level deep, с предметными именами и оглавлением для длинных файлов.

## Правила проведения eval

1. **Один eval = одна сессия.** Между запусками `/clear` для чистоты эксперимента
2. **Фиксировать всё.** Токены, время, результат, наблюдения — в `docs/researches/eval-assets/`
3. **Минимум 3 повтора.** Каждый eval case проводится минимум 3 раза для учёта дисперсии
4. **Результат → MEMORY.md.** После завершения eval сохранить вывод в memory проекта
5. **Обновить recommendation type.** Результат eval может повысить или понизить тип рекомендации в `summary.md`

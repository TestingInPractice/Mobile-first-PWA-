# Методология литературного обзора

> **Дата:** 2026-05-09
> **Версия:** 1.0
> **Назначение:** Единый методологический слой для оценки источников и формулирования рекомендаций в `docs/`

---

## 1. Evidence Strength Taxonomy

Система оценки источников расширяет базовую шкалу L1/L2/L3 (см. `.claude/rules/research-sources.md`) дополнительным измерением **Evidence Strength** — силой доказательства.

### Категории Evidence Strength

| Категория | Маркер | Описание | Примеры |
|-----------|--------|----------|---------|
| **Peer-reviewed** | `[peer-reviewed]` | Статья прошла независимое рецензирование на конференции (ICLR, NeurIPS, ICML, EMNLP, AAAI, IJCNLP) или в журнале (TACL, Nature, ACM, Computational Linguistics) | Liu et al. TACL 2024, Harada et al. ICLR 2025 |
| **Preprint** | `[preprint]` | Статья на arXiv или аналогичном репозитории, не прошедшая рецензирование. Может содержать ценные данные, но выводы не проверены независимыми экспертами | arXiv:2506.16411, arXiv:2601.15300 |
| **Vendor docs** | `[vendor]` | Официальная документация, технические отчёты и engineering-блоги вендоров (Anthropic, OpenAI, Google, Meta) | Anthropic context engineering, OpenAI structured outputs |
| **Community measurement** | `[community]` | Блоги, статьи, бенчмарки от независимых специалистов и компаний. Данные могут быть полезны, но методология не стандартизирована | Arize AI SWE-bench, Particula Tech prompt length |
| **Local eval** | `[local-eval]` | Локальные эксперименты dot_ai: сравнение вариантов навыков, rules, workflow на реальных задачах проекта | См. `local-eval-plan.md` |

### Соотношение с уровнями L1/L2/L3

| L-уровень | Включает | Evidence Strength внутри уровня |
|-----------|----------|-------------------------------|
| **L1** | Рецензируемые статьи + preprints | `[peer-reviewed]` или `[preprint]` |
| **L2** | Официальная документация вендоров | `[vendor]` |
| **L3** | Блоги, статьи, сообщество | `[community]` |
| — | Локальные эксперименты dot_ai | `[local-eval]` |

**Правило:** при первичном mentioning источника указывается L-уровень и Evidence Strength. При последующих ссылках достаточно L-уровня.

```
Пример: "Curse of Instructions" `[L1]` `[peer-reviewed]` — при первом упоминании
        Curse of Instructions `[L1]` — при последующих ссылках
```

## 2. Правила интерпретации источников

### 2.1. Что разрешено для принципов vs гипотез

| Категория | Можно использовать для | Нельзя использовать для |
|-----------|----------------------|------------------------|
| `[peer-reviewed]` | Принципы (`hard recommendation`), общие закономерности | — |
| `[preprint]` | Гипотезы (`hypothesis`), эвристики (`heuristic`), направления для локального eval | Универсальных рекомендаций без caveat |
| `[vendor]` | Рекомендации, специфичные для конкретного инструмента/модели | Универсальных закономерностей, независимых от вендора |
| `[community]` | Гипотезы, направления для проверки | Численных рекомендаций без пометки `[community]` и caveat |
| `[local-eval]` | Практические рекомендации для dot_ai | Общих утверждений о поведении LLM |

### 2.2. Source laundering — запрещённые паттерны

**Source laundering** — неявное повышение силы источника при пересказе:

1. **L3 → универсальное правило:** blog-измерение (например, «sweet spot 500–2,000 токенов») пересказано как «оптимальный размер — 500–2,000 токенов» без указания источника. Правильно: «sweet spot 500–2,000 токенов `[L3]` `[community]` — community measurement, применимость к конкретному проекту требует локального eval»
2. **Preprint → установленный факт:** нерецензированная статья цитируется как «доказано, что...». Правильно: «предварительные данные показывают... `[L1]` `[preprint]`»
3. **Vendor-специфичное → универсальное:** рекомендация Anthropic для Claude пересказана как общая закономерность. Правильно: «для Claude (Anthropic) — ..., универсальность для других моделей не проверена `[L2]` `[vendor]`»
4. **Единичное измерение → порог:** одно blog-измерение превращается в жёсткий порог. Правильно: указать sample size, модель, условия, пометить `[community]` или `[local-eval]`

### 2.3. Формулировки, связанные с Evidence Strength

| Формулировка | Допустима при | Недопустима при |
|-------------|--------------|----------------|
| «Исследования показывают» / «Evidence-based» | `[peer-reviewed]` или `[local-eval]` | `[preprint]`, `[community]` |
| «Предварительные данные» / «Есть основания полагать» | `[preprint]`, `[vendor]`, `[community]` | — |
| «Рекомендуется» (без caveats) | `[peer-reviewed]`, `[local-eval]`, `[vendor]` для конкретного инструмента | `[community]`, `[preprint]` |
| «Эвристика» / «Практическое правило» | Любой источник, если явно помечен | — |
| «Модель не может» | Только `[peer-reviewed]` с прямым доказательством | `[preprint]`, `[community]` |
| «Растёт риск / нужен eval» | Любой источник | — |

## 3. Evidence Scope для численных утверждений

Все численные рекомендации в `docs/` должны сопровождаться **Evidence Scope** — областью применимости.

### Шаблон Evidence Scope

```
[Evidence Scope: model={модель}, date={дата}, benchmark={бенчмарк/задача}, source={статус источника}]
```

### Примеры

| Утверждение | Evidence Scope |
|-------------|---------------|
| Деградация 13.9-85% при росте контекста | `[Evidence Scope: GPT-4, Claude, дата: 2025, benchmark: math/QA/coding, source: peer-reviewed]` |
| Sweet spot 500-2,000 токенов | `[Evidence Scope: unspecified model, дата: 2025, benchmark: Q&A юридические документы, source: community]` |
| ~12K токенов overhead на субагента | `[Evidence Scope: Claude Code, дата: 2025, benchmark: internal measurement, source: vendor]` |
| Overhead 5-7 правил в навыке | `[Evidence Scope: GPT-4, Claude, дата: 2025, benchmark: multi-instruction following, source: peer-reviewed]` |
| Частота галлюцинаций 3-4% | `[Evidence Scope: GPT-4, Claude Sonnet, дата: 2024, benchmark: Vectara HHEM summarization, source: vendor]` |

### Когда Evidence Scope обязателен

- Процентные изменения (`15x`, `73-124%`, `90.2%`)
- Абсолютные числа (`500 строк`, `12K токенов`, `5-7 правил`)
- Сравнительные утверждения (`X лучше Y`, `рост/снижение`)
- Пороговые значения (`sweet spot`, `maximum`, `optimal`)

### Когда Evidence Scope не нужен

- Общие знания (syntax, стандартные API)
- Определения и терминология
- Описания архитектуры и паттернов без численных claims

## 4. Типы рекомендаций

Каждая практическая рекомендация в `docs/` относится к одному из трёх типов:

### 4.1. Hard Recommendation `[hard]`

**Определение:** Рекомендация, основанная на peer-reviewed исследованиях или локальных eval с воспроизводимыми результатами. Может применяться как общее правило.

**Требования к источникам:**
- Минимум один источник `[peer-reviewed]` или воспроизводимый `[local-eval]`
- Численные данные с Evidence Scope
- Непротиворечивость с другими peer-reviewed источниками

**Пример:** «Критичные инструкции размещать в начале и конце промпта `[hard]` — поддерживается Lost in the Middle `[L1]` `[peer-reviewed]`»

### 4.2. Heuristic `[heuristic]`

**Определение:** Практическое правило, подтверждённое vendor-документацией, preprint-ами или community-наблюдениями. Полезно как отправная точка, но требует проверки для каждого конкретного случая.

**Требования к источникам:**
- Минимум один источник `[vendor]` или `[preprint]`
- Или согласованные наблюдения из нескольких `[community]` источников
- Указание границ применимости

**Пример:** «Тело навыка: 500-1,500 токенов `[heuristic]` — sweet spot по community measurements `[L3]` и engineering blog данным `[L2]`»

### 4.3. Hypothesis `[hypothesis]`

**Определение:** Предположение, основанное на ограниченных данных или аналогиях. Требует проверки через local eval или дополнительного исследования.

**Требования к источникам:**
- Может основываться на единичном `[community]` наблюдении
- Или на аналогии из смежной области
- Или на предварительных `[preprint]` данных с ограниченной выборкой

**Пример:** «Навык с LEARNINGS.md retrieval превосходит статический SKILL.md `[hypothesis]` — логически обосновано, но не проверено eval»

### 4.4. Правила повышения типа рекомендаций

```
hypothesis → heuristic: подтверждено минимум двумя независимыми источниками или одним [vendor] + согласованность с [peer-reviewed]
heuristic → hard: подтверждено [peer-reviewed] исследованием или воспроизводимым [local-eval]
```

## 5. Правила для обновления существующих документов

1. **Не переписывать всё сразу.** При редактировании документа добавлять Evidence Strength и Recommendation Type к новым и изменённым утверждениям. Существующие `[L1]`/`[L2]`/`[L3]` маркеры сохраняются.

2. **Численные claims без Evidence Scope не удалять.** Вместо этого добавить caveat: `[community measurement, модель/дата неизвестны, применимость к dot_ai требует проверки]`

3. **Формулировки «модель не может» заменить** на «растёт риск» или «нужен eval», если источник не `[peer-reviewed]` с прямым доказательством.

4. **Библиография:** новые источники добавляются с пометкой Evidence Strength. Существующие записи при пересмотре уточняются: добавляется `[peer-reviewed]` или `[preprint]`.

5. **Новые hard recommendations** требуют: (а) минимум один источник `[peer-reviewed]` с прямым доказательством, ИЛИ (б) воспроизводимый `[local-eval]` (3+ повтора), ИЛИ (в) источник `[vendor]` + согласованность с `[peer-reviewed]` (только для инструмент-специфичных рекомендаций). Рекомендации, не соответствующие критериям, публикуются как `[heuristic]` или `[hypothesis]` с указанием gaps. Это правило предотвращает накопление неподтверждённых универсальных рекомендаций.

## 6. Чеклист для проверки документа

| # | Критерий | ✅ / ❌ |
|---|----------|---------|
| 1 | Каждое нетривиальное утверждение имеет маркер L-уровня | |
| 2 | Численные claims имеют Evidence Scope или caveat | |
| 3 | Практические рекомендации типизированы (`[hard]`, `[heuristic]`, `[hypothesis]`) | |
| 4 | Нет community-derived чисел без пометки `[community]` | |
| 5 | Нет формулировок «модель не может» без `[peer-reviewed]` обоснования | |
| 6 | Preprint-источники не используются для универсальных рекомендаций | |
| 7 | Vendor-специфичные данные не представлены как универсальные | |
| 8 | Библиография сгруппирована по L-уровням | |
| 9 | Новые `[hard]` рекомендации соответствуют критериям из §5, правило 5 | |

## 7. Research Backlog

Приоритизированный список направлений для углубления литобзора. Детальный roadmap с решениями dot_ai и источниками для поиска — в `docs/best_practice/summary.md` §5.4.

| # | Направление | Приоритет | Текущее покрытие | Ключевой gap | Решения dot_ai, зависящие от направления |
|---|-------------|-----------|------------------|--------------|----------------------------------------|
| 1 | Instruction Following: пределы многоинструктивности | **P0** | Curse of Instructions `[L1]`, IFEval/SIFo | Количественные пороги compliance для 2026 frontier-моделей; взаимодействие instructions в skills и rules | Размер CLAUDE.md, количество rules, структура навыков |
| 2 | Agentic Coding Evals | **P0** | SWE-bench `[L1]`, RepoBench `[L1]` | Agent-specific eval данные; dependency utilization; сравнение agent-agnostic vs agent-specific | Eval стратегия, архитектура spec-implementer |
| 3 | Context Engineering: long vs retrieved | **P1** | Lost in the Middle `[L1]`, Li et al. RAG vs Long `[L1]` | Пороги для codebase exploration (10–50 файлов); evidence-first влияние на разные задачи | Архитектура субагентов, навыки с evidence-first |
| 4 | Multi-Agent Systems: оркестрация | **P1** | AgentBench `[L1]`, ChatDev/MetaGPT/CAMEL `[L1]` | Single-agent vs multi-agent для задач spec-implementer уровня; коммуникационные overhead данные | Архитектура субагентов, порог перехода к сложным схемам |
| 5 | Skill Evolution / Memory | **P2** | Reflexion `[L1]`, Adaptive Memory Distillation `[L1]` | Долгосрочный эффект learning logs; порог prompt bloat; eval-gated self-modification | LEARNINGS.md в навыках, механизм дистилляции |
| 6 | Governance / Requirements | **P2** | IEEE 29148 `[L2]`, RARE `[L1]` | Spec-driven vs prompt-driven эмпирика; ADR с evidence-based обоснованиями; traceability для AI-кода | spec-creation workflow, критерии качества spec |

### Правила ведения backlog

1. **При добавлении нового источника** — проверить, закрывает ли он gap в backlog. Если да — обновить «Текущее покрытие» и понизить приоритет или закрыть направление
2. **При изменении архитектуры dot_ai** — проверить, не появились ли новые зависимости от незакрытых направлений
3. **Раз в квартал** — ревью backlog: повысить приоритет направлений, от которых зависят планируемые изменения; понизить приоритет устаревших
4. **Новые направления** — добавлять с приоритетом P2, повышать до P1/P0 только если от них зависит запланированное архитектурное решение

# CLAUDE.md — структура и конфигурация справочного файла проекта

> **Дата:** 2026-05-07
> **Версия:** 1.0
> **Источник:** [rules.md](rules.md) (эффективность инструкций), [skills.md](skills.md) (progressive disclosure)
> **Библиография:** [summary.md](summary.md#54-библиография)

---

CLAUDE.md — главный конфигурационный файл проекта для Claude Code. Загружается **eager** (при каждой сессии), переживает compaction и определяет базовое поведение AI-ассистента. Практическое руководство по настройке: какие секции включать, как позиционировать контент, каких ошибок избегать.

## 1.1 Обязательные секции и порядок

CLAUDE.md не имеет формальной схемы — модель интерпретирует текст как структурированный промпт. Однако эмпирически выработан порядок секций, максимизирующий compliance `[L2]` `[L3]`.

**Рекомендуемый порядок секций:**

| # | Секция | Назначение | Приоритет позиционирования |
|---|--------|-----------|---------------------------|
| 1 | **Название и описание** | Что это за проект, в 1–2 предложениях | Начало (высокий recall) |
| 2 | **Workflow** | Обязательная последовательность шагов | Начало (критичное правило) |
| 3 | **Идеология** | Принципы и ограничения проекта | Начало–середина |
| 4 | **Структура приложения** | Дерево файлов с назначением | Середина (справочный) |
| 5 | **Ключевые технологии** | Стек, версии, назначение | Середина (справочный) |
| 6 | **Конфигурация** | .env, переменные, точки входа | Середина (справочный) |
| 7 | **Навыки (Skills)** | Таблица доступных навыков | Середина (справочный) |
| 8 | **Документация** | Ссылки на артефакты | Середина (справочный) |

**Обоснование порядка** `[L1]` `[L2]`:
- **Начало** (секции 1–2): название + workflow — критичные правила, которые модель должна соблюдать всегда. Lost in the Middle: наивысший recall в начале контекста
- **Середина** (секции 3–7): идеология, структура, технологии — важный, но не критичный контекст. Модель обращается к нему по мере необходимости
- **Конец**: если есть критичные напоминания (например, «не коммить без запроса»), продублировать в последних строках — конец промпта имеет наивысший recall `[L1]`

**Примеры секций из реальных проектов:**

```
## Workflow
┌─────────────────┐     ┌────────────────────┐     ┌──────────────────┐
│  plan-release   │────►│ implement-spec-stage│────►│ integrate-release│
│  ① ТЗ + Ветка   │     │ ② Реализация       │     │  ③ Мерж          │
└─────────────────┘     └────────────────────┘     └──────────────────┘
```

```
## Структура приложения
src/
├── agents/           ← AI-агенты
│   ├── orchestrator/  # Оркестратор задач
│   └── tracker/       # Трекер-агент
├── shared/           ← Общие DTO и контракты
└── infrastructure/   ← Внешние интеграции
```

```
## Ключевые технологии
| Технология | Версия | Назначение |
|---|---|---|
| PydanticAI | 0.2 | Фреймворк агентов |
| SQLAlchemy | 2.0 | ORM |
```

## 1.2 Размер и плотность

CLAUDE.md — единственный файл, загружаемый **всегда** и **полностью**. Его размер напрямую влияет на compliance `[L1]`.

**Количественные рекомендации:**

| Метрика | Рекомендация | Обоснование |
|---------|-------------|-------------|
| **Строки** | ≤200 строк | Читаемость для разработчика, обзор без скролла `[L3]` `[community]` |
| **Токены** | 500–2,000 (sweet spot) | Оптимальная производительность, наивысшая точность `[L2]` `[L3]` `[Evidence Scope: unspecified model, дата: 2025, benchmark: community Q&A tests, source: community + vendor]` |
| **Максимум** | ≤4,000 токенов | Верхний предел перед активной деградацией `[L3]` `[community]` `[Evidence Scope: community measurement, модель не указана, дата: 2025, source: community]` |
| **Поведенческие правила** | 5–7 ключевых | Curse of Instructions: больше правил → хуже compliance `[L1]` `[peer-reviewed]` `[Evidence Scope: frontier-модели, дата: 2025, benchmark: ICLR 2025, source: peer-reviewed]` |

**Практический подход к сжатию:**
- Таблицы вместо списков — компактнее, модель парсит лучше `[L2]`
- ASCII-диаграммы вместо текстового описания workflow — одна диаграмма заменяет 10 строк
- Ссылки на внешние файлы вместо inline-контента — ARCHITECTURE.md, AGENT_DESIGN.md

**Зоны производительности** `[L2]` `[L3]`. **Примечание:** численные границы — community measurement, не универсальные пороги `[heuristic]`:

| Зона | Токены | Поведение |
|------|--------|-----------|
| Sweet spot | 500–2,000 | Оптимальная производительность |
| Убывающая отдача | 2,000–4,000 | На 40-80% медленнее `[Evidence Scope: unspecified model, дата: 2025, benchmark: single project Q&A, source: community measurement]`, минимальный прирост |
| Активная деградация | 4,000+ | Снижение качества, пропущенные детали |

## 1.3 Позиционирование критичного контента

U-образная кривая Lost in the Middle: наивысший recall в начале и конце контекста, провал в середине `[L1]`.

**Стратегия позиционирования для CLAUDE.md:**

```
┌─────────────────────────────────────────────────────────┐
│ НАЧАЛО (высокий recall)                                 │
│ • Название проекта и назначение                         │
│ • Workflow: обязательная последовательность шагов       │
│ • Критичные ограничения («один шаг за сессию»)          │
├─────────────────────────────────────────────────────────┤
│ СЕРЕДИНА (сниженный recall)                             │
│ • Структура проекта (дерево файлов)                     │
│ • Технологии и конфигурация                             │
│ • Ссылки на документацию                                │
├─────────────────────────────────────────────────────────┤
│ КОНЕЦ (высокий recall)                                  │
│ • Критичные напоминания                                 │
│ • Формат вывода / коммитов                              │
│ • Next skill после завершения                            │
└─────────────────────────────────────────────────────────┘
```

**Примеры позиционирования:**

1. **Workflow в начале** — модель видит обязательную последовательность при каждом обращении к CLAUDE.md `[L2]`
2. **Дерево файлов в середине** — справочный контекст, загружается по мере необходимости `[L3]`
3. **«Не коммить без запроса» в конце** — критичное ограничение в позиции высокого recall `[L1]`

## 1.4 Иерархия CLAUDE.md

Claude Code поддерживает иерархию CLAUDE.md-файлов с каскадным наследованием `[L2]`.

**Механизм загрузки:**
- Корневой `CLAUDE.md` — загружается **всегда**, переживает compaction
- Поддиректория `sub/dir/CLAUDE.md` — загружается **лениво**, при работе с файлами в этой директории `[L2]`
- `@import`-паттерн — ссылка на внешний файл из CLAUDE.md, рекурсивно до 5 уровней вложенности `[L2]`
- Claude обходит директории **снизу вверх** от cwd, загружая каждый найденный CLAUDE.md `[L2]`

**Приоритизация (от высокого к низкому):**

| Уровень | Файл | Когда загружается | Переживает compaction |
|---------|------|-------------------|----------------------|
| Enterprise | Managed policy | Всегда | Да |
| CLI args | `--system-prompt` | Всегда | Да |
| Project | `/CLAUDE.md` | Всегда | Да |
| User | `~/.claude/CLAUDE.md` | Всегда | Да |
| Subdirectory | `src/CLAUDE.md` | При работе в `src/` (lazy) | Нет |
| Imported | `@ARCHITECTURE.md` | При Read tool call | Нет |

**@import-синтаксис:**

```
// В CLAUDE.md:
Смотри @.ai/ARCHITECTURE.md для деталей архитектуры.
Личные настройки: @~/.claude/my-preferences.md
```

- Поддерживает относительные и абсолютные пути
- `@~/.claude/...` — персональные импорты (без ошибки у коллег без файла) `[L3]`
- Не вычисляется внутри code blocks (кодовых блоков) `[L2]`
- Рекурсия до 5 прыжков `[L2]`

**Рекомендации по иерархии:**
- **Корневой CLAUDE.md**: ≤200 строк, только критичные правила и workflow. Не перегружать справочным контентом
- **Поддиректории**: для больших монорепозиториев — директория-специфичные инструкции (например, `frontend/CLAUDE.md` с React-конвенциям)
- **@import**: для справочных файлов, которые не нужны при каждой сессии (ARCHITECTURE.md, DATA_MODELS.md)
- **Не дублировать**: если правило есть в корневом CLAUDE.md, не повторять в поддиректории `[L2]`

**Compaction и CLAUDE.md:**
- Корневой CLAUDE.md **переживает compaction** — система перечитывает его с диска после сжатия и реинжектирует `[L2]` `[L3]`
- Это делает CLAUDE.md «наиболее надёжным способом дать Claude постоянные инструкции» `[L3]`
- Rules с `paths:` frontmatter тоже сохраняются, но поддиректория-специфичные CLAUDE.md — нет
- Custom compaction prompt (`compactPrompt` в settings.json) позволяет указать, что обязательно сохранить при сжатии `[L3]`

## 1.5 Анти-паттерны

**1. Дублирование с rules и skills** `[L2]`:
- Правила, уже вынесенные в `.claude/rules/` или SKILL.md, повторённые в CLAUDE.md → рост контекста, противоречия
- Правило: CLAUDE.md = общие принципы, rules = контекстные инструкции, skills = процедуры

**2. Слишком абстрактные правила** `[L2]`:
- «Пиши чистый код» vs «Используй Conventional Commits: type(scope): message»
- Конкретные, измеримые правила эффективнее абстрактных

**3. Отсутствие структуры** `[L2]` `[L3]`:
- Сплошной текст без заголовков → модель не может парсить границы секций
- XML-теги или Markdown-заголовки помогают модели выделять секции

**4. Перегрузка справочным контентом** `[L3]`:
- Полная документация API в CLAUDE.md → активная деградация (>4,000 токенов)
- Справочный контент — в ARCHITECTURE.md, DATA_MODELS.md, README.md

**5. Workflow без визуализации** `[L2]`:
- Текстовое описание шагов «Сначала сделай X, потом Y...» — модель теряет последовательность
- ASCII-диаграмма или таблица с ENTRY/EXIT — однозначная спецификация

## 1.6 Кросс-инструментальное сравнение

CLAUDE.md — конфигурационный файл проекта для Claude Code. Аналогичные механизмы существуют в других AI-ассистентах.

| Инструмент | Файл | Загрузка | Иерархия | Особенности |
|---|---|---|---|---|
| **Claude Code** | `CLAUDE.md` | Eager, всегда | Root + поддиректории + `@import` | Переживает compaction. Markdown-формат `[L2]` |
| **Cursor** | `.cursorrules` (deprecated), `.cursor/rules/*.mdc` | Always / Auto (glob) / Agent-Requested | Always > Auto > Agent-Requested | Frontmatter: `description`, `globs`, `alwaysApply`. Лимит <500 строк `[L2]` |
| **GitHub Copilot** | `.github/copilot-instructions.md`, `.github/instructions/*.instructions.md` | Всегда (repo-wide), при совпадении glob (path-specific) | Personal > Repository > Organization | `applyTo:` для conditional loading. Non-deterministic conflict resolution `[L2]` |
| **Aider** | `CONVENTIONS.md` (через `--read`) | При старте сессии | Нет иерархии | Read-only файл в контексте, нет path-gating `[L2]` |
| **Windsurf/Cline** | `.windsurfrules`, `.clinerules/*.md` | Всегда или conditional | Workspace > Global | Лимит 6,000 символов/файл (Windsurf). Cline: кросс-инструментальная совместимость `[L2]` |

**Общие паттерны:**
- **Markdown-формат** — все инструменты используют Markdown или YAML frontmatter
- **Conditional loading** — паттерн «загружать только релевантные инструкции» доминирует (glob, applyTo, paths)
- **Контекстный бюджет** — primary constraint. Больше контента ≠ лучше `[L1]` `[L2]`
- **Нет формальных merge-семантик** — при конфликте между файлами поведение не определено `[L2]`

**Агностичный минимум:**
Для проектов, использующих несколько AI-ассистентов, общий знаменатель:
- `AGENTS.md` — распознаётся Copilot, Cline и другими инструментами
- `.cursorrules` → `.cursor/rules/` — для Cursor
- `CLAUDE.md` — для Claude Code
- `CONVENTIONS.md` — для Aider
- Дублирование минимально: общие правила → AGENTS.md, инструмент-специфичные →各自 файлы

## 1.7 Чеклист качества CLAUDE.md

Пошаговая проверка для нового или существующего CLAUDE.md:

1. **≤200 строк** — файл читается без скролла, в sweet spot токенов
2. **Название проекта** в первых 2 строках — что это, зачем
3. **Workflow** в первых 20 строках — обязательная последовательность шагов
4. **ASCII-диаграмма или таблица** для workflow — визуальная однозначность
5. **≤7 поведенческих правил** — Curse of Instructions compliance
6. **Структура проекта** — дерево файлов с назначением каждого каталога
7. **Таблица технологий** — стек, версии, назначение каждого компонента
8. **Нет дублирования с rules/skills** — общее в CLAUDE.md, контекстное в rules, процедурное в skills
9. **Справочный контент вынесен** — ARCHITECTURE.md, DATA_MODELS.md, README.md — отдельно
10. **Критичные напоминания в конце** — дублирование ключевых ограничений в последних строках
11. **Нет абстрактных правил** — каждое правило конкретное и измеримое
12. **Markdown-структура** — заголовки, таблицы, списки — модель парсит лучше, чем сплошной текст

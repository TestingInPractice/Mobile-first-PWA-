# Architecture, Memory и вспомогательные файлы

> **Дата:** 2026-05-07
> **Версия:** 1.0
> **Источник:** [rules.md](rules.md) (Lost in the Middle), [skills.md](skills.md) (lazy loading)
> **Библиография:** [summary.md](summary.md#54-библиография)

---

Вспомогательные файлы проекта — ARCHITECTURE.md, CHANGELOG.md, Templates, Memory, Specs — справочный контекст, загружаемый выборочно. Не входят в каждый запрос (в отличие от CLAUDE.md), но предоставляют критичную информацию при необходимости.

## 1.1 ARCHITECTURE.md

ARCHITECTURE.md — документ архитектуры проекта, загружаемый через `@import` или Read tool. Даёт AI моделированное «понимание» структуры проекта.

**Рекомендуемая структура:**

```markdown
# Архитектура {Project Name}

## Назначение
1–2 предложения: что это за проект и его основная цель.

## Точка входа
Путь к главному файлу/модулю и краткое описание потока выполнения.

## Структура модулей
Дерево каталогов с назначением каждого (ASCII-диаграмма).

## Ключевые паттерны
3–5 архитектурных решений: naming conventions, dependency flow, error handling.

## Потоки данных
Как данные проходят через систему (от входа до выхода).
```

**Аннотации к секциям:**

| Секция | Обязательная? | Что писать | Что НЕ писать |
|--------|-------------|-----------|--------------|
| **Назначение** | Да | 1–2 предложения о сути проекта | Подробное описание бизнес-логики |
| **Точка входа** | Да | Путь к entry point + что происходит при запуске | Код инициализации |
| **Структура модулей** | Да | ASCII-дерево + 1 строка на каталог | Содержимое каждого файла |
| **Ключевые паттерны** | Да | 3–5 паттернов с 1-строчным описанием | Детали реализации |
| **Потоки данных** | Рекомендуется | Направление данных: вход → обработка → выход | Все промежуточные шаги |

**Размер:** ≤100 строк. ARCHITECTURE.md — справочный файл, не документация API. Модель загружает его через Read tool, значит — занимает контекстный бюджет `[L2]` `[L3]`.

**Пример (минимальный):**

```markdown
# Архитектура dot_ai

## Назначение
Централизованный репозиторий правил, навыков и шаблонов для AI-assisted разработки.

## Точка входа
init/SKILL.md → навык установки dot_ai в целевой проект

## Структура модулей
dot_ai/
├── rules/          ← Формальные правила (→ секции CLAUDE.md)
├── skills/         ← Навыки-шаблоны с {{PLACEHOLDER}}
├── templates/      ← Шаблоны артефактов проекта
└── init/           ← Навык установки

## Ключевые паттерны
- Template-based distribution: навыки = шаблоны с плейсхолдерами
- Workflow pipeline: обязательная последовательность с ENTRY/EXIT
- Source of truth: все изменения в dot_ai → init → целевой проект

## Потоки данных
dot_ai (source) → init/SKILL.md → целевой проект (/.claude/, /CLAUDE.md)
```

## 1.2 CHANGELOG.md

CHANGELOG.md — история изменений проекта в формате Keep a Changelog. Привязан к версионированию SemVer `[L2]`.

**Формат:**

```markdown
# CHANGELOG

## [X.Y.Z] — YYYY-MM-DD

### Added — {Краткое описание}
Описание что добавлено.

**Документы:**
- `path/to/file.md`: описание файла

**Spec:** `.ai/specs/{branch-name}.md`
**Tests:** описание тестирования

## [X.Y.Z] — YYYY-MM-DD
...
```

**Секции записи:**

| Секция | Когда использовать |
|--------|-------------------|
| `Added` | Новый функционал, документы, фичи |
| `Changed` | Изменения в существующем поведении |
| `Deprecated` | Функционал, который будет удалён |
| `Removed` | Удалённый функционал |
| `Fixed` | Исправления багов |
| `Security` | Исправления уязвимостей |

**Рекомендации:**
- Каждая запись привязана к spec: `**Spec:** .ai/specs/{branch}.md`
- Обратный хронологический порядок (новые — сверху)
- Версия определяется по SemVer: major/minor/patch bump
- Описание содержит: что изменилось, какие файлы, ссылку на spec

## 1.3 Templates

Templates — шаблоны артефактов проекта, используемые при init-установке. Хранятся в `templates/`.

**Организация:**

```
templates/
├── CLAUDE.md.tpl              # Основной конфиг проекта
├── CHANGELOG.md.tpl           # История изменений
├── ARCHITECTURE.md.tpl        # Архитектура
├── settings.json.tpl          # Claude env config
├── settings.local.json.tpl    # Permissions
└── deploy.env.example.tpl     # Переменные для деплоя
```

**Placeholder-конвенции:**

| Placeholder | Описание | Пример значения |
|-------------|----------|----------------|
| `{{PROJECT_NAME}}` | Имя проекта (lowercase, underscore) | `agent_tracker` |
| `{{PROJECT_DISPLAY}}` | Читаемое название | `Agent Tracker` |
| `{{PROJECT_DESCRIPTION}}` | Назначение (1–2 предложения) | `AI pipeline для проектного управления` |
| `{{SRC_PACKAGE}}` | Python-пакет в `src/` | `agent_tracker` |
| `{{STACK}}` | Основные технологии | `PydanticAI, SQLAlchemy` |
| `{{TECH_TABLE}}` | Таблица технологий | Markdown-таблица |
| `{{DATE}}` | Дата создания | `2026-05-07` |
| `{{PACKAGE_MANAGER}}` | Менеджер пакетов | `uv` |
| `{{TEST_RUN_CMD}}` | Команда запуска тестов | `uv run pytest` |
| `{{DOCKER_SERVICE}}` | Docker-сервис | `agent-tracker-bot` |
| `{{ENTRY_POINT}}` | Точка входа | `src/agent_tracker/main.py` |
| `{{MODULE_STRUCTURE}}` | Дерево модулей | ASCII-дерево |

**Минимальный набор шаблонов:**
1. `CLAUDE.md.tpl` — обязательный, основной конфиг
2. `settings.json.tpl` — конфигурация модели
3. `settings.local.json.tpl` — permissions
4. `ARCHITECTURE.md.tpl` — архитектура (пустой, заполняется вручную)
5. `CHANGELOG.md.tpl` — начальная запись

## 1.4 Decisions (ADR)

Decisions — журнал архитектурных решений проекта в формате MADR (Markdown Architectural Decision Records). Каждое решение = отдельный файл с контекстом, альтернативами и обоснованием `[L2]`.

**Зачем:** архитектурные решения внутри specs теряются при архивации. ADR обеспечивает кросс-фичевую видимость и эволюцию решений `[L2]`.

**Когда добавлять:** когда проект накапливает >5 решений, пересекающих границы отдельных specs.

**Структура хранения:**

```
docs/decisions/
├── 0001-record-architecture-decisions.md
├── 0002-use-python-fastapi.md
└── 0003-postgresql-over-mongodb.md
```

**Структура записи (MADR 4.0, минимальный шаблон):**

```markdown
# {короткое название}

## Context and Problem Statement
{2–3 предложения: контекст и проблема}

## Considered Options
* {вариант 1}
* {вариант 2}

## Decision Outcome
Chosen option: "{вариант}", because {обоснование}.

### Consequences
* Good, because {положительное последствие}
* Bad, because {отрицательное последствие}
```

**Статусная модель:**

| Статус | Значение |
|--------|----------|
| Proposed | Решение предложено, обсуждается |
| Accepted | Решение принято к реализации |
| Deprecated | Решение устарело, но сохраняется для истории |
| Superseded | Решение заменено другим (указать номер ADR) |

**Первое ADR:** решение использовать ADR для трекинга архитектурных решений — фиксирует саму практику `[L2]`.

**Правила:**
- Нумерация последовательная: `NNNN-title-with-dashes.md`
- Не удалять — даже отменённые решения сохраняют ценность
- Хранить рядом с кодом в Git, не в wiki/Confluence `[L2]`
- Связывать с spec при наличии: `**Spec:** .ai/specs/{branch}.md`

**Исследование:** [governance-patterns.md](../researches/governance-patterns.md)

## 1.5 Specs

Specs — технические задания в формате IEEE 29148:2018, расположенные в `.ai/specs/`.

**Naming convention:**
- Новая фича: `feature-v{version}-{slug}.md` (например, `feature-v0.0.4-reference-files-config.md`)
- Багфикс: `bugfix-v{version}-{slug}.md`
- Ветка: `feature/v{version}-{slug}` или `bugfix/v{version}-{slug}`

**Структура spec:**

```markdown
# Feature Title — Feature Specification (vX.Y.Z)

> **Дата создания:** YYYY-MM-DD
> **Ветка:** feature/vX.Y.Z-slug
> **Статус:** 🟡 В работе
> **IEEE 29148:2018 Score:** N/100

---

## Workflow выполнения
| Шаг | Действие | Навык | Статус |
| а | Создание ТЗ | — | ✅ |
| б | Реализация | implement-spec-stage | ⬜ |
| в | Мерж | integrate-release | ⬜ |

## Видение
Пробел, цель, целевая аудитория.

## Архитектурные решения
Таблица: вопрос → решение.

## Этапы реализации
### Этап 1: Название ⬜
Файлы, что делаем, определение готовности.

---

## IEEE 29148:2018 Validation
Оценка по критериям.
```

**Архивирование:** после merge spec перемещается в `.ai/specs/archive/` навыком integrate-release.

## 1.6 Чеклист

1. **ARCHITECTURE.md ≤100 строк** — назначение, точка входа, модули, паттерны, потоки данных
2. **5 секций ARCHITECTURE.md** — назначение, точка входа, структура, паттерны, потоки данных
3. **CHANGELOG.md в Keep a Changelog** — Added/Changed/Fixed секции, обратный хронологический порядок
4. **Каждая CHANGELOG-запись привязана к spec** — `**Spec:** .ai/specs/{branch}.md`
5. **Templates с {{PLACEHOLDER}}-конвенциями** — uppercase, descriptive, min 5 шаблонов
6. **Specs с IEEE 29148:2018** — score ≥85, количественные критерии готовности
7. **Архивирование specs** — после merge → `.ai/specs/archive/`
8. **Decisions (опционально)** — `docs/decisions/NNNN-*.md` в формате MADR, когда решений >5
9. **Первое ADR = решение использовать ADR** — фиксирует саму практику
10. **Не удалять ADR** — даже отменённые решения сохраняют ценность
